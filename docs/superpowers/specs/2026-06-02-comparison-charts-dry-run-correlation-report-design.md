# Comparison Charts, Flow Dry Run, and Correlation Report Design

## Goal

Add stronger run comparison exports, add one-step flow dry-run debugging, and fix missing correlation IDs in exported reports without changing normal run behavior.

## Current Context

Singular Excel reports are generated by `ExcelGenerator`. They stream detailed logs, collect time buckets, and create a `Charts` sheet with response time, percentiles, RPS, active threads, bandwidth, response time by endpoint, and RPS by endpoint.

Comparison Excel reports are generated by `RunComparisonExcelGenerator`. They currently write `Compare Summary`, `Endpoint Delta`, `Run A Logs`, and `Run B Logs`, but no chart dashboard.

Normal flow execution creates a `RunEntity`, builds a `FlowExecutionContext`, and runs asynchronously through `FlowAsyncRunner` and `FlowExecutor`. Endpoint nodes execute real endpoints through `EndpointExecutorFactory` and currently enqueue request logs for non-JS endpoints.

Export mapping in `RunServiceImpl` already copies `RequestLogEntity.correlationId` into report DTOs, so the correlation bug needs export-level regression coverage and a targeted fix at the first broken link found during implementation.

## Requirements

### Comparison Charts

The comparison export must include a `Charts` sheet similar to the singular report. Each chart must plot both runs in one chart, using distinct colors and clear legend labels.

Required charts:

- Response Time Over Time
- Response Time Percentiles Over Time
- Requests Per Second Over Time
- Active Threads Over Time
- Response Bandwidth Over Time
- Response Time by Endpoint
- RPS by Endpoint

The implementation should preserve the existing summary, endpoint delta, and per-run detail log sheets.

### Flow Dry Run

The app must support running exactly one selected flow step for debugging. Dry run must execute real endpoint calls for endpoint steps, including HTTP, GRPC, JDBC, JS, and plugin-backed executors.

Dry run must not create persisted runs and must not persist request logs. Temporary variables returned from a dry-run step are kept separately from normal flow variables. A later dry-run step can use them, but normal flow execution must not read them unless the user explicitly copies them into normal run variables.

The dry-run response should include:

- Executed step ID and type
- Next step ID selected by routing logic
- Output data from the node handler
- Updated temporary variables after pre-processors and post-processors
- Any endpoint result/debug information already available from the node handler path
- Error details when the selected step fails

### Correlation ID Reports

Excel and HTML detailed report exports must populate the correlation ID column/field when request logs have a correlation ID. The fix must include tests that fail if correlation ID is dropped during export.

## Architecture

### Backend

Add dry-run command/response DTOs and a `FlowService.dryRunStep` method exposed by `FlowController`.

Add dry-run execution support that reuses existing flow step lookup, environment resolution, processor execution, and `FlowNodeHandler` dispatch. The context should be configured with:

- No `RunEntity`
- A generated transient run ID only for debug metadata if needed
- A generated correlation ID
- `persistRequestLogs=false`
- Variables initialized from active environment, request variables, and supplied temporary dry-run variables

Add a log-persistence guard to `EndpointNodeHandler` through `FlowExecutionContext`, so dry run executes real endpoint calls but does not enqueue or publish request logs.

For comparison charts, extend `RunComparisonExcelGenerator` with chart data collection during `appendRunDetailRow`. Use the same bucket calculations and chart layout as the singular generator, but write series for both runs into each chart. The hidden chart-data sheet should remain hidden.

### Frontend

Add dry-run state to the flow/workspace provider that stores temporary variables by flow ID. Add a selected-step dry-run action in the flow editor. The UI should show response/output, next step ID, and the current temporary variable snapshot. Normal run dialogs should remain unchanged except for an optional explicit copy action if implemented.

### Data Flow

Comparison export:

1. `RunServiceImpl.exportRunComparison` builds both `RunReport`s.
2. It streams Run A logs and Run B logs.
3. `RunComparisonExcelGenerator` writes detail rows and collects per-run time buckets.
4. `export` finalizes the chart sheet and writes the workbook.

Dry run:

1. Frontend sends selected `stepId`, environment override, and temporary variables.
2. Backend loads the flow and configured steps.
3. Backend builds a non-persisting `FlowExecutionContext`.
4. Backend runs one step through the normal handler path.
5. Backend returns selected route, output data, error info, and updated variables.
6. Frontend stores updated temporary variables for the next dry-run step only.

## Error Handling

Dry run should return normal API errors for missing flow, missing step, missing active environment, and invalid configuration. Execution failures inside endpoint handlers should follow current endpoint-node behavior and return failure output instead of persisting logs.

Comparison chart generation should skip the `Charts` sheet only when both runs have no detailed logs. Empty or sparse series should produce zero values rather than throwing.

## Testing

Backend tests:

- Comparison Excel export creates `Charts` with the same seven line charts as singular reports.
- Comparison chart legends include Run A and Run B labels.
- Dry run executes one endpoint step and does not save a `RunEntity` or request log.
- Dry run carries temporary variables forward in the response.
- Excel detailed logs include correlation ID.
- HTML detailed logs include correlation ID.

Frontend tests:

- Dry-run provider state stores temporary variables separately from normal run variables.
- Dry-run action calls the backend with selected step ID and temporary variables.

## Out of Scope

This change does not add multi-step dry-run automation, breakpoints, persisted debug sessions, or report chart customization.
